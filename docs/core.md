
## Общая структура

Модуль `core/` содержит всю вычислительную часть виртуального измерительного комплекса: движок пошагового выполнения, базовые классы объектов, конкретные модели и генераторы, конфигурации экспериментов и вспомогательные утилиты. Ядро полностью отделено от пользовательского интерфейса, что позволяет запускать симуляцию в автономном режиме или с любым GUI.

Состав модуля:
* `simulation_base/` – движок и базовый класс объектов.
* `elements/` – генераторы сигналов, преобразователи (модели, фильтры) и пассивные инспекторы.
* `experiments/` – сборочные классы, создающие и связывающие объекты для конкретных экспериментов.
* `data/` – таблицы параметров аккумулятора.
* `utils/` – математические утилиты (билинейная интерполяция).

## simulation_base – движок и основа объектов

### SimulationEngine

Центральный диспетчер симуляции. Хранит зарегистрированные объекты и определяет порядок их выполнения.

**Принцип работы:**
- При создании каждый `SimulationObject` автоматически регистрируется в движке и получает уникальный числовой идентификатор.
- Порядок вызовов `solve_frame()` соответствует порядку регистрации.
- Движок предоставляет метод `get_result(target_id, tail_len)`, который возвращает данные другого объекта с учётом того, выполнился ли он в текущем фрейме. Если цель уже выполнена, `tail_len` увеличивается на 1, чтобы гарантировать чтение предыдущего кадра и избежать использования «будущих» данных. Это основа двойной буферизации.
- Счётчик `_frames_count` и `dt` (шаг по времени) позволяют отслеживать модельное время.

**Принятые решения:**
- Жёсткий порядок регистрации даёт детерминированное поведение без необходимости топологической сортировки графа зависимостей.
- Автокоррекция `tail_len` обеспечивает атомарность данных в пределах одного кадра: никакой объект не может «подсмотреть» ещё не готовый результат текущего фрейма.

### SimulationObject

Абстрактный предок всех компонентов. Определяет интерфейс, управление историей и механизм сбора входных данных с автоматической временной синхронизацией.

**Ключевые механизмы:**

1. **История результатов** – ограниченная очередь `deque` фиксированного размера. Каждый кадр завершается вызовом `_push_result()`, который сохраняет словарь с выходными данными.

2. **Связывание входов** – метод `set_input(input_key, source_id, output_key)` задаёт правило, откуда брать данные для конкретного входа.

3. **Автоматическая ретроспектива** – наиболее важное архитектурное решение.  
   - Каждый объект имеет `_retrospective_shift` – личное смещение во времени (в кадрах). Изначально 0.  
   - При вызове `solve_frame()` движком сначала выполняется `_collect_inputs()`, который для каждого входа вызывает `_get_input()`.  
   - В `_get_input()`, если хотя бы один из объектов (вызывающий или источник) имеет ретроспективу > 0, вместо движка используется прямая формула `effective_tail = 1 + retro_self - retro_source`. Это позволяет объектам, живущим в прошлом, видеть согласованные данные.  
   - Если данных недостаточно (история источника слишком мала), выбрасывается `RetrospectiveAdjustment`. Обёртка `solve_frame()` перехватывает это исключение, увеличивает `_retrospective_shift` на 1 и пропускает кадр. На следующем фрейме объект попробует снова.  
   - Таким образом, система сама выравнивается по причинно-следственным связям без ручного задания задержек. EKF автоматически получает ретроспективу 2, сумматор – 1, батарея – 1.

4. **Сбор входов** – `_collect_inputs()` заполняет словарь `_collected_inputs`, который затем используется в `_solve_frame()`. Это избавляет наследников от необходимости вызывать `_get_input()` вручную.

**Маркерные классы:**
- `Generator` – только выдаёт данные (нет входов).
- `Transformator` – принимает входы, преобразует, выдаёт результат.
- `Inspector` – пассивно читает данные других объектов (например, DataLogger).

## elements – компоненты симуляции

### Генераторы (generators/)

Генераторы не имеют входов и на каждом фрейме записывают результат в историю.

- **ConstantGenerator** – выдаёт одно и то же значение. Полезен для задания постоянных тока, температуры, опорных сигналов.
- **CyclicProfileGenerator** – циклически воспроизводит заданный массив значений. Поддерживает `profile_scale` (количество кадров на одно значение) и `linear_change` (плавная линейная интерполяция между соседними точками профиля). Используется для реалистичных профилей тока и температуры.
- **NoiseGenerator** – генерирует гауссовский шум с заданными `sigma` и `bias`. Может использоваться для имитации погрешностей датчиков.
- **ParameterGenerator** – хранит значение, которое можно менять извне, и записывает его в историю.
- **StatefulGenerator** – позволяет задать произвольное правило обновления состояния, полезен для таймеров, счётчиков, пилообразных сигналов.

### Трансформаторы (transformers/)

Принимают входы от других объектов (через `_collected_inputs`), выполняют вычисления и выдают результаты.

- **BatteryThevenin1RC** – модель аккумулятора по схеме Тевенина с одним RC-звеном и гистерезисом OCV.  
  Использует табличную интерполяцию параметров `R0, R1, C1` по SOC и температуре.  
  Выходы: `voltage_terminal`, `soc`, `vc`, `ocv`, `hysteresis_dyn`, `current`, `temperature`.

- **EKFSOCEstimator** – расширенный фильтр Калмана для оценки SOC и поляризационного напряжения.  
  *Прогноз*: используются матрицы `A` и `B`, построенные на основе текущих `R1, C1`. Реализована строгая линеаризация – вычисляются производные `R1` и `C1` по SOC (конечные разности) и строится полная матрица Якоби `F`.  
  *Коррекция*: предсказанное напряжение сравнивается с измеренным, вычисляется инновация. Якобиан измерения `H` также включает производную `R0` по SOC.  
  *Ковариации*: поддерживается загрузка плоских ключей `Q00, Q11, P00, P11` из JSON с автоматическим преобразованием в диагональные матрицы.  
  *Гистерезис*: мгновенная составляющая вычисляется как `-s * sign(I)` (знак исправлен относительно батареи для соответствия эксперименту). Динамическая составляющая получается от батареи.

- **NoiseAdderTransformator** – сумматор, складывающий сигнал и шум. Используется для формирования «измеренных» значений тока, напряжения и температуры.

### Инспекторы (inspectors/)

- **DataLogger** – пассивно записывает историю указанных объектов в CSV.  
  Заголовок формируется один раз при первом сбросе буфера на основе всех ключей, присутствующих в накопленных строках. Это гарантирует полный набор колонок, несмотря на постепенное появление данных при разогреве симуляции. Поддерживает подпапки в имени файла (`filename = "default/record.csv"`).

## experiments – конфигурации экспериментов

Классы `SimulationSetup` отвечают за создание и связывание объектов для конкретного сценария.

- **SimulationSetup** – базовый класс, требующий реализации метода `build()`.
- **BatterySimulationSetup** – создаёт генераторы тока и температуры, модель батареи и связывает их входы. Простейший эксперимент без алгоритмов оценки.
- **EKFSimulationSetup** – основной сетап для экспериментов с EKF.  
  *Создаёт*: генераторы тока и температуры (константные или циклические), модель батареи, три генератора шума и три сумматора, EKF, DataLogger.  
  *Подключает EKF* к зашумлённым данным (ток, напряжение, температура).  
  *Параметры* `current_mode`, `current_profile`, `temperature_mode`, `temperature_profile`, шумы передаются через конструктор и могут быть переопределены JSON-конфигами.  
  *Использует* `AppContext.WithContext` для создания объектов, что позволяет автоматически загружать параметры из файлов.

## data – таблицы параметров

Файл `param_tables.py` содержит:
- Сетки SOC и температуры.
- Таблицу `R0` (омическое сопротивление) в явном виде.
- Опорные значения `R1` и `C1` при 25 °C, взятые из литературы.
- Функции `compute_R1` (закон Аррениуса с энергией активации 35 кДж/моль) и `compute_C1` (слабая линейная зависимость) для пересчёта на другие температуры.
- Автоматическую генерацию полных матриц `R1_VALUES` и `C1_VALUES` по сетке.
- Итоговый словарь `DEFAULT_PARAM_TABLES`, передаваемый в конструкторы модели батареи и EKF.

Это решение позволяет легко модифицировать параметры без изменения кода моделей.

## Заключение

Модуль `core/` реализует детерминированное, расширяемое ядро симуляции с автоматической временной синхронизацией. Чёткое разделение на генераторы, трансформаторы и инспекторы, а также унифицированный механизм сбора входов делают архитектуру прозрачной и удобной для добавления новых компонентов.
