
## Общая структура

Модуль `apps/` содержит классы приложений — точки входа для различных режимов работы измерительного комплекса. Каждое приложение наследуется от `BaseSimulationApp` и задаёт свою конфигурацию симуляции и интерфейс. Общая функциональность (движок, управление симуляцией, сценарии, окружение) вынесена в базовый класс, что исключает дублирование.

Состав модуля:
* `base_simulation_app.py` — базовое приложение с движком, GUI, сценариями.
* `battery_simulation_app.py` — симуляция только батареи без алгоритмов оценки.
* `ekf_soc_app.py` — основной эксперимент с EKF SOC Estimator.
* `noise_simulation_app.py` — тестирование генератора шума.
* `utils/config.py` — глобальный контекст `AppContext` и система иерархических JSON-конфигураций.

## Базовая инфраструктура (`BaseSimulationApp`)

Базовый класс реализует всё, что не зависит от конкретного эксперимента.

**Управление движком:**
Создаётся экземпляр `SimulationEngine`. Методы `step`, `start`, `pause`, `reset` управляют пошаговым выполнением объектов.

**Цикл GUI:**
Периодический вызов `update_funcs` (собранных через `TreeParser`) для обновления виджетов. Метод `_frame_callback` запускает симуляцию с настраиваемой скоростью (`simulation_speed`), что позволяет ускорять или замедлять эксперимент.

**Окружение приложения:**
Свойства `env_path`, `records_path`, `logs_path`, `scenarios_path`, `settings_path` формируют пути относительно `app_name`. Метод `setup_environment` создаёт папки. Это изолирует данные каждого приложения.

**Настройки:**
Главный файл `settings.json` загружается и сохраняется через `load_settings`/`save_settings`. Метод `_apply_main_settings` применяет параметры `dt`, `speed`, `autostart`, `duration_frames`. Автозапуск и остановка по длительности позволяют проводить полностью автоматические эксперименты.

**Сценарии:**
Метод `apply_scenario` копирует JSON-файлы из папки сценария (`scenarios/<name>/`) в рабочую `settings/`, замещая базовые настройки. Это позволяет быстро переключаться между конфигурациями без правки кода.

**Порядок запуска (`run`):**
1. Создаётся окружение.
2. Загружаются главные настройки.
3. Если активирован сценарий (`use_scenario: true`), файлы копируются, настройки перечитываются.
4. Применяются параметры симуляции.
5. Инициализируется `AppContext`.
6. Создаётся GUI, запускается симуляция (если `autostart`).
Такой порядок гарантирует, что `AppContext` видит окончательную рабочую папку, а сценарий может переопределить всё, включая главные настройки.

## Точка входа (`main.py`)

Лаунчер, выбирающий приложение:
* Аргумент командной строки: `python main.py ekf`.
* Интерактивный ввод, если аргумент не указан.
* Поддержка флагов `--docs-reboot` и `--settings-reboot` для принудительного обновления документации или сброса настроек (управляется через `AppContext`).

## Конкретные приложения

### NoiseSimulationApp
* Создаёт `NoiseGenerator`.
* UI: слайдеры `sigma` и `bias`, график истории, гистограмма, теоретическая PDF.
* Используется для отладки генератора шума.

### BatterySimulationApp
* Создаёт `BatterySimulationSetup` с генераторами тока/температуры и моделью батареи.
* UI: истинные значения напряжения, SOC, тока при заданном профиле нагрузки.
* Полезно для проверки модели и таблиц параметров.

### EKFSOCApp (основное)
* Создаёт `EKFSimulationSetup` через `AppContext.WithContext`, что позволяет переопределять параметры из JSON.
* Включает: генераторы тока и температуры (константные/циклические), модель батареи, шумы, сумматоры, EKF, DataLogger.
* UI: вкладки Control, EKF SOC (графики, Live Parameters, настройки EKF), Recording, Settings.

## Система конфигурирования (`AppContext`)

Класс `AppContext` связывает объекты с конфигурационными файлами.

**Реестр:** хранит список `(объект, путь, load_settings)`. Путь формируется как `parent@child`.

**`WithContext`:**
* Принимает родительский объект, класс и ключевое слово.
* Формирует путь.
* Загружает JSON из `settings/{путь}.json`, переопределяя аргументы конструктора.
* Создаёт объект и регистрирует его.
* Если файла нет, создаёт пустой JSON и, при необходимости, генерирует `[docs].md` с аннотациями параметров (управляется флагами `is_force_reboot_docs` и `is_force_reboot_settings`).

**Транзитные объекты:** при `load_settings=False` объект регистрируется, но JSON не загружается. Это используется для структурных узлов (например, `EKFUISetup`).

**Принятые решения:**
1. **Плоская структура файлов с иерархическими именами** – вместо вложенных папок используется `@` как разделитель. Упрощает навигацию и копирование.
2. **Автоматическая документация параметров** – при создании JSON-файла генерируется `[docs].md` с аннотациями из сигнатуры `__init__` и docstring. Это решает проблему забытых описаний параметров.
3. **Флаги принудительного обновления** – позволяют перегенерировать документацию или сбросить настройки без удаления файлов вручную.

## Заключение

Модуль `apps/` реализует гибкую и расширяемую архитектуру: базовый класс предоставляет общую функциональность, а конкретные приложения фокусируются на специфике эксперимента. Система `AppContext` и сценариев делает настройку полностью декларативной и воспроизводимой. Автоматическая генерация документации параметров снижает порог входа для новых пользователей.
