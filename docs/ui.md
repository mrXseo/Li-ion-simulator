
## Общая структура

Модуль `ui/` содержит весь код графического интерфейса, построенного на DearPyGui. Архитектура интерфейса основана на двух ключевых концепциях:

1. **Декларативное построение** – интерфейс описывается в виде дерева объектов `UINode`, которые затем рекурсивно инициализируются и обновляются через `TreeParser`. Это позволяет собирать сложные многоуровневые интерфейсы из готовых блоков без ручного управления тегами и иерархией.

2. **Переиспользуемые виджеты** – базовый класс `BaseWidget` и его наследники (`HistoryPlotWidget`, `ParameterControlWidget`, `DataLoggerWidget` и др.) инкапсулируют логику создания и обновления элементов DPG. Каждый виджет имеет чёткий жизненный цикл: `__init__` → `init(parent_tag)` → `build()` → периодический `update()`.

Интерфейс полностью отделён от ядра симуляции и взаимодействует с ним только через объекты `SimulationObject`, что позволяет запускать симуляцию как с GUI, так и без него.

Состав модуля:
* `tools/` – инфраструктура декларативного UI (`UINode`, `TreeParser`).
* `widgets/` – переиспользуемые виджеты.
* `setups/` – компоновки интерфейса для конкретных приложений.

## tools – декларативное построение интерфейса

### UINode

Узел дерева интерфейса, соответствующий одному элементу или контейнеру DearPyGui.

**Поля:**
- `name` – логическое имя узла (используется для формирования тега).
- `tree_type` – тип узла: `DPG_COMPLEX_COMPONENT` (контейнер: окно, группа, вкладка), `DPG_SIMPLE_COMPONENT` (простой виджет: кнопка, текст), `WIDGET` (кастомный виджет, наследник `BaseWidget`).
- `init_func` – функция, создающая DPG-элемент. Принимает `tag`, `parent` и `**kwargs`.
- `build_func` – опциональная функция, вызываемая после построения всех элементов (используется для пост-настройки: установка колбэков, захват тегов).
- `update_func` – опциональная функция, вызываемая на каждом кадре для обновления содержимого.
- `childrens` – словарь дочерних узлов.

**Принятые решения:**
- Разделение на `init_func` и `build_func` позволяет сначала создать все элементы интерфейса, а затем выполнить действия, требующие знания полных тегов (например, сохранение тега для последующего обновления). Это решает проблему «позднего связывания» тегов в DearPyGui.
- `update_func` собираются `TreeParser` в единый список и вызываются из главного цикла приложения, что даёт централизованное обновление всех виджетов.

### TreeParser

Статический класс для работы с деревом `UINode`.

**Основные методы:**
- `tree_init(root, snowball_tag, **extra_params)` – рекурсивно инициализирует все узлы, конкатенируя имена в `snowball_tag` для формирования полного тега DPG.
- `tree_build(root, snowball_tag)` – рекурсивно вызывает `build_func` для каждого узла.
- `get_update_list(root)` – собирает все `update_func` в плоский список для периодического вызова.
- `str_format(root)` – возвращает псевдографическое строковое представление дерева (удобно для отладки).
- `find(...)` – поиск узлов по имени с поддержкой контекстных предков.

**Вспомогательная функция:**
- `widget_to_node(name, widget)` – преобразует экземпляр `BaseWidget` в `UINode`, автоматически создавая `init_func` и `build_func`, которые вызывают соответствующие методы виджета.

**Принятые решения:**
- Конкатенация имён в `snowball_tag` гарантирует уникальность тегов без необходимости задавать их вручную.
- `get_update_list` работает рекурсивно, поэтому можно добавлять `update_func` на любом уровне дерева, и они будут автоматически найдены.

## widgets – переиспользуемые виджеты

### BaseWidget

Базовый класс для всех виджетов. Определяет жизненный цикл и общие атрибуты.

- `tag` – уникальный идентификатор виджета (будет использован как тег корневого элемента).
- `simulation_object` – ссылка на объект симуляции, с которым связан виджет.
- `parent_tag` – тег родительского контейнера (устанавливается при `init`).

**Методы:**
- `init(parent_tag)` – создаёт все DPG-элементы.
- `build()` – пост-настройка после построения.
- `update()` – периодическое обновление данных.

**Принятые решения:**
- Разделение на `init` и `build` позволяет сначала создать все элементы, а затем настроить их (например, установить колбэки слайдеров, начальные пределы осей). Это особенно важно в декларативном подходе, где элементы создаются рекурсивно и родительские теги становятся известны только после `tree_init`.

### HistoryPlotWidget

Линейный график истории значений из `SimulationObject`.

- Поддерживает несколько серий (ключей) с пользовательскими легендами.
- Ось X показывает «кадры назад» (0 – текущий, –1 – предыдущий).
- Настраиваемый `window_size` (количество отображаемых точек).
- Опциональные фиксированные пределы оси Y (`y_limits`).

**Использование:** `HistoryPlotWidget(tag="...", simulation_object=battery, data_keys={'soc': 'True SOC'})`

### ParameterControlWidget

Универсальный виджет управления параметрами объекта.

- Поддерживает **обычный режим**: слайдер + опциональное поле ввода.
- Поддерживает **научный режим** (`use_mantissa=True`): слайдер мантиссы + опциональный ввод экспоненты.
- Флаг `lazy` добавляет кнопку «Set» и отображение текущего значения объекта. При `lazy=True` значение применяется к объекту только по нажатию кнопки, что предотвращает непрерывную отправку изменений при движении слайдера.
- Флаг `extra_accuracy` добавляет поле ввода экспоненты для точного задания малых величин (например, `1e-12`).

**Принятые решения:**
- Научный режим был добавлен, потому что стандартные слайдеры DearPyGui не позволяют точно задавать значения порядка `1e-12`. Разделение на мантиссу и экспоненту решает эту проблему без создания отдельных виджетов.
- Режим `lazy` необходим для EKF, где частые изменения ковариаций могут сбивать фильтр. Пользователь может настроить параметры и применить их одним нажатием.

### DataLoggerWidget

Виджет управления записью данных.

- Кнопка Start/Stop для включения/выключения записи.
- Кнопка Flush Now для принудительной записи буфера в файл.
- Отображение статистики: Written rows, Saved rows, Unsaved rows.
- Отображение списка целей (targets).

### HistoryPlotHistogramWidget

Гистограмма квантованных значений из истории. Зависит от `HistoryPlotWidget` для получения данных и синхронизации осей. Используется в `NoiseSetup` для визуализации распределения шума.

### NoiseDistributionWidget

Виджет управления параметрами гауссовского шума. Содержит слайдеры `sigma` и `bias`, а также график теоретической плотности вероятности (PDF). Используется в `NoiseSetup`.

## setups – компоновки интерфейса

Каждый сетап наследуется от `BlankSetup` и реализует метод `get_setup()`, возвращающий корневой `UINode` для соответствующего интерфейса.

### BlankSetup

Базовый класс для всех сетапов. Хранит `tag` – префикс для тегов элементов.

### EKFUISetup

Основной интерфейс эксперимента с EKF. Реализует двухколоночный макет с TabBar'ами.

**Структура:**
- **Вкладка Models settings:**
  - Строка 1: график истинного SOC | панель Live Parameters.
  - Строка 2: график оценки EKF SOC | настройки ковариаций EKF.
- **Вкладка Error statistic:** график ошибки SOC.
- **Вкладка Covariance:** график P00 и P11.
- **Вторая строка (Other plots):** таб-бар с графиками истинного и измеренного напряжения, инновации.

**Вспомогательные фабрики:** `_make_group`, `_make_child_window`, `_make_tab_bar`, `_make_tab`, `_make_text_header` – сокращают дублирование кода при построении дерева.

**Интеграция с LiveParameters:** создаются `ParameterControlWidget` для тока, температуры и шумов, которые передаются в `LiveParametersSetup` и отображаются внутри соответствующих вкладок. Это решение позволило собрать все основные органы управления в одном компактном блоке, не загромождая основной интерфейс.

### LiveParametersSetup

Панель мониторинга и управления ключевыми параметрами.

- **Селектор вкладок:** Current, Voltage, Temperature, Resistance, SOC.
- **Содержимое вкладок:**
  - Current: слайдеры тока и шума тока, истинное/измеренное значение, график.
  - Voltage: слайдер шума напряжения, истинное/измеренное напряжение, инновация, график.
  - Temperature: слайдеры температуры и шума температуры, значения, график.
  - Resistance: R0, R1, C1.
  - SOC: истинный SOC, оценка, ошибка, ёмкость от температуры.
- **Мини-графики** создаются через `HistoryPlotWidget` с высотой 250 и шириной 450 пикселей.

**Принятые решения:**
- Все вкладки строятся декларативно через `UINode` и `widget_to_node`. Это гарантирует корректное управление видимостью без ручного скрытия/показа групп.
- Слайдеры управления током, температурой и шумами передаются извне через конструктор. Это позволяет `EKFUISetup` создать их и передать в `LiveParametersSetup`, сохраняя единый источник истины для параметров.

### SettingsUISetup

Редактор JSON-конфигураций с двухпанельным интерфейсом.

- **Левая панель:** дерево объектов, построенное по данным из `AppContext.get_configurable_objects()`. Узлы формируются по иерархии `@`. Каждый конфигурируемый объект отображается как `selectable` с индикатором изменений.
- **Правая панель:** динамический редактор. Для каждого ключа загруженного JSON создаётся подходящий виджет (`checkbox` для `bool`, `input_float` для чисел, `input_text` для строк). Цветной индикатор показывает, изменён ли параметр относительно сохранённого значения.
- **Кнопки:** Save, Reload.
- **Плашки:** предупреждение о несохранённых изменениях и о необходимости перезапуска после сохранения.

**Принятые решения:**
- Использование `build_func` для захвата тега правой панели и текстовых полей – позволяет безопасно обновлять содержимое после построения.
- Индикаторы изменений (`|` зелёный/жёлтый) дают мгновенную обратную связь о том, какие параметры были отредактированы.
- Автоматическое создание `[docs].md` при первом сохранении конфига (через `AppContext.WithContext`) решает проблему документирования параметров.

### SimulationControlSetup

Панель управления симуляцией:
- Кнопки Step, Start, Pause, Reset.
- Слайдер скорости симуляции.
- Текст с текущим модельным временем.
- График загрузки процессора.

### Остальные сетапы

- **BatteryUISetup** – интерфейс для симуляции только батареи: слайдеры тока и температуры, графики напряжения, SOC, тока.
- **NoiseSetup** – интерфейс для тестирования генератора шума: график истории, гистограмма, распределение, текущее значение шума.
- **ConstantSetup, CyclicProfileSetup, ParameterSetup, StatefulSetup** – простые интерфейсы для отдельных генераторов.

## Заключение

Модуль `ui/` реализует гибкую и расширяемую архитектуру пользовательского интерфейса на основе декларативного описания деревом `UINode`. Чёткое разделение на переиспользуемые виджеты и сетапы позволяет легко модифицировать существующие интерфейсы и создавать новые. Интеграция с ядром симуляции осуществляется исключительно через объекты `SimulationObject`, что сохраняет независимость вычислительной части от GUI.
